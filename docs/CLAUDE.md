# Ziro Designer documentation style guide

These are the conventions for writing and editing the Ziro Designer docs, so
every page keeps one consistent voice no matter who drafts it. The team writes
and reviews these docs with the help of AI tools, and owns the result. The rules
below are our house style.

## Punctuation and typography

Use a plain hyphen for joins (for example, "browser-native"). Do not use the em
dash or the en dash anywhere on the site. Standardizing on the hyphen keeps
punctuation uniform across authors and tools and removes the em/en/hyphen
ambiguity that otherwise creeps in. When a sentence wants a pause, use a comma, a
colon, parentheses, or two shorter sentences.

## Voice

Write plainly and directly, for an engineer who wants to get something done.
Prefer short words. Use sentence case for headings.

## Code and examples

Keep code samples minimal and runnable. Show the command, then say what it does.

## This repository is public

Do not put secrets, API keys, tokens, or private URLs in docs, config, comments,
or commits. Assume everything here is world-readable and permanent.
